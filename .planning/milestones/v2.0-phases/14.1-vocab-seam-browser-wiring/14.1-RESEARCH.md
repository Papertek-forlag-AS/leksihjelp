# Phase 14.1: Vocab-Seam Browser Wiring - Research

**Researched:** 2026-04-25
**Domain:** Integration wiring -- vocab-seam.js + spell-check.js gap closure
**Confidence:** HIGH

## Summary

The milestone audit identified two integration blockers preventing 9 spell-check rules from firing in the browser, despite all fixture suites passing at 100%. The root cause is a seam gap: `vocab-seam-core.js:buildIndexes()` returns 8 indexes added by Phases 8-14, but `spell-check.js` never wires them into the `vocab` object it assembles at lines 225-246. Rules guard with `if (!index) return []`, so they pass fixtures (which call `buildIndexes()` directly) but are completely silent in the browser.

A secondary blocker is the accidental deletion of `doc-drift-de-address.js` from the working tree (still committed at HEAD). This causes 3 release gates to fail: `check-explain-contract`, `check-rule-css-wiring`, and `check-benchmark-coverage`.

**Primary recommendation:** Add 7 missing getters to `vocab-seam.js`, add 8 missing wiring lines to `spell-check.js` vocab object, restore the deleted file, and verify all release gates exit 0.

## Architecture Patterns

### The Three-Layer Vocab Pipeline

The extension uses a three-layer architecture for vocab data:

```
vocab-seam-core.js      (Layer 1: Pure index builder -- Node + browser)
       |
       v  buildIndexes() returns full state object with ALL indexes
       |
vocab-seam.js           (Layer 2: Browser IIFE, chrome.storage, getters)
       |
       v  self.__lexiVocab.get*() individual getter methods
       |
spell-check.js          (Layer 3: Consumer, assembles vocab object for rules)
       |
       v  const vocab = { ... } at lines 225-246
       |
spell-rules/*.js        (Layer 4: Rules read ctx.vocab.*)
```

**The gap is between Layer 2 and Layer 3.** New indexes from Phases 8-14 are:
- Correctly built in Layer 1 (buildIndexes returns them)
- Partially exposed in Layer 2 (only `getParticipleToAux` has a getter)
- Completely absent from Layer 3 (spell-check.js vocab object)

### Existing Getter Pattern (vocab-seam.js)

All getters follow the same pattern. Example from line 299:

```javascript
getParticipleToAux: () => (state && state.participleToAux) ? state.participleToAux : new Map(),
```

For Map-valued indexes, the fallback is `new Map()`.

### Existing Wiring Pattern (spell-check.js lines 225-246)

All wiring follows the same pattern:

```javascript
const vocab = {
  nounGenus:        VOCAB.getNounGenus(),
  // ... each key matches what rules read from ctx.vocab.*
  participleToAux:  VOCAB.getParticipleToAux(),   // <-- this line is MISSING
};
```

### Rule Guard Pattern

All affected rules guard with the same pattern:

```javascript
const participleToAux = ctx.vocab && ctx.vocab.participleToAux;
if (!participleToAux || participleToAux.size === 0) return [];
```

When the key is `undefined` (not wired), `ctx.vocab.participleToAux` is `undefined`, the guard fires, and the rule returns `[]` silently.

## Common Pitfalls

### Pitfall 1: Getter Naming Convention
**What goes wrong:** Naming the getter `getParticiple_to_aux` when the state key is `participleToAux` and the rule reads `ctx.vocab.participleToAux`.
**How to avoid:** The getter name must be `get` + PascalCase of the state key. The wiring key in spell-check.js must exactly match what the rule reads from `ctx.vocab.*`.

### Pitfall 2: Missing the participleToAux Getter
**What goes wrong:** Assuming all 8 getters need to be added to vocab-seam.js.
**How to avoid:** `getParticipleToAux` already exists at line 299. Only the spell-check.js wiring is missing for it. The other 7 getters need both the getter AND the wiring.

### Pitfall 3: Forgetting frSubjonctifDiffers
**What goes wrong:** Treating the FR subjonctif wiring as only 2 indexes (presens + forms) when it actually needs 3 (+ differs).
**How to avoid:** fr-subjonctif.js reads 3 separate indexes. The `frSubjonctifDiffers` Map is the homophony guard -- without it the rule silently exits.

### Pitfall 4: Not Restaging After File Restore
**What goes wrong:** Running `git checkout HEAD -- doc-drift-de-address.js` but not verifying the file actually appears on disk.
**How to avoid:** After restore, verify with `ls -la` and run the 3 failing gates.
