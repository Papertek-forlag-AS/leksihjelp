## Decisions Made

### Validator Scope: Conservative Predictor

Initial broad predictor (all Norwegian adjective neuter rules) surfaced 171 "defects" — most false positives from legitimate data that my predictor rules misjudged. Narrowed the predictor to emit predictions ONLY for the monosyllabic-pure-vowel class; all other classes return null → entry skipped. Result: 12 defects surfaced (the real target class × 2 languages), 0 false positives. The predictor's header documents this design intent so future editors understand the narrowness is intentional, not a TODO.

### NN Generator Investigation: Independent, Requires Matching Fix

Verdict: **NN adjectives are independently generated** by `enrichAdjectiveNN()` in `generate-nn-lexicon.js:1336`. The function has its own declension branch at line 1393-1397 (default: `noytrum = w + 't'`) — exactly the same defect as the NB generator. Fix applied symmetrically: same mono-vowel branch + same compound allowlist. Future runs of `npm run generate:nn-lexicon` will preserve the `-tt` forms rather than regenerate `blåt`.

The `_metadata.description` in `vocabulary/lexicon/nn/manifest.json` says NN is "derived from Bokmål via NB→NN transformation" — but the transformation derives the WORD (nbToNnWord applies lexical replacements + spelling rules), then the NN generator independently computes declension via `enrichAdjectiveNN`. So the NN side isn't a copy of NB's generator output; it has its own generator with its own bug.

### fin_adj Schema: Matches NB Verbatim, No Silent Fallback

Grep-inspection of existing NN adjectives (`god_adj`, `pen_adj`, `stor_adj`) confirmed NN uses the same declension keys as NB: `maskulin/feminin/noytrum/flertall/bestemt`. NN-specific differences are in the comparison forms (`-are/-ast` vs NB `-ere/-est`) and the `meir` vs `mer` periphrastic. Authored `fin_adj` NN with `finare/finast/finaste`; NB with `finere/finest/fineste`. Post-write Node inline verification confirmed `noytrum === 'fint'` before commit.

Plan's Blocker 3 constraint ("silent NB fallback not acceptable") honored: the NN entry is grep-verified against NN's own schema, not blind-copied from NB.

### Diphthong-Guard Regex Refinement (Rule-1 Deviation)

Plan's Task 2 Step 2 committed verbatim to `w.length <= 3 && /[aeiouyæøå]$/.test(w)` for the mono-vowel branch. Testing standalone showed this regex would regress `høy` → `høytt` (3 chars, `y` in the vowel set → fires). `høy` and `gøy` exist as real NB adjectives with CORRECT pre-existing neuter `høyt` / `gøyt` (diphthong + `-t`, not doubled). Added Rule-1 refinement: second clause `!/[aeiouyæøå][aeiouyæøå]$/.test(w)` rejects words whose final TWO letters are both vowels (diphthongs). Post-refinement: `ny` → `nytt` ✓, `høy` → `høyt` ✓, `brei` → `breit` ✓.

This is a genuine data-correctness deviation — the plan's regex would have shipped a regression. Committed with clear note in the Task 2 commit body.

### Compound Allowlist Over Compound Regex

A general "compound ending in mono-vowel" regex `[bcdfghjklmnpqrstvwxzæøå][aeiouyæøå]$` matches the 3 intended compounds (himmelblå, mørkeblå, lyseblå) — but also matches invariable loan-word adjectives (`rosa`, `lilla`, `ekstra`, `sjalu`, `sporty`, `moro`), over-doubling them to `rosatt` / `ekstratt` / etc. Switched to an explicit allowlist `(blå|grå|rå|ny|fri|bra|tro|sky)$` that covers known compound-forming mono-vowel adjective roots. If a new mono-vowel root lands in the lexicon, the allowlist is a one-word addition.

### Hand-Edit Over Regenerate-All

Running `npm run enrich:nb` to apply the generator fix would also regenerate declension for all 443 NB adjectives. Inspecting that run's diff showed it introduces 384+ unrelated line changes, some of which are regressions (himmelblått → himmelblåt because the compound rule wasn't originally there; blågrønt → blågrønn because the `-nn` double-consonant branch is wrong in the generator; barnevennlig → barnevennligt because the `-lig` branch adds `-t`). These are pre-existing generator quality issues — fixing them is out of Gap A scope. **Decision**: apply the 6 mono-vowel neuter fixes as hand-edits; keep the generator fix for future-proofing (so no one re-introduces the defect); do NOT re-run the generator against the data in this plan.

## Issues Encountered

### Pre-existing generator quality issues surfaced (out of Gap A scope)

Re-running `npm run enrich:nb` to apply the mono-vowel generator fix was found to also regenerate declension/comparison forms for 443 NB adjectives, producing 384+ line changes in the bank. Spot-inspection of those changes revealed MULTIPLE pre-existing generator bugs beyond the mono-vowel target:

These are genuine data-correctness concerns but are explicitly OUT of Gap A scope (which is scoped to the mono-vowel class + blå_adj). Documented here as deferred follow-up; not addressed in this plan. They exist in pre-plan state and will continue to exist until a future audit targets them.
