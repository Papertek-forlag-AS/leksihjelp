## Deviations from Plan

### Auto-fixed Issues (during Task 3 human-verify smoke test)

All four bugs surfaced during user-driven browser testing — classic human-verify gate catching what the fixture suite couldn't.

**1. [Rule 1 - Bug] Register badge missing from popover header**
- **Found during:** Task 3 (Chrome smoke test — user asked "which register is this rule checking?")
- **Issue:** Popover showed rule title + copy but no indication of whether the rule was evaluating against NB or NN document register. Causality confusing for dialect-mix flags.
- **Fix:** Added `Bokmål`/`Nynorsk` badge in popover header alongside rule title. Reads `vocab.getLanguage()` at render time.
- **Files modified:** `extension/popup/popup.js`, `extension/content/floating-widget.js` (popover DOM assembly paths)
- **Verification:** User re-tested after rebuild — badge visible, shows correct register
- **Commit:** `a2d34e4`

**2. [Rule 1 - Bug] Feature-gated lookup indexes silent modal-verb on `kan gikk`**
- **Found during:** Task 3 (smoke test scenario — typed `kan gikk` in NB textarea, expected flag, got nothing)
- **Issue:** Modal-verb lookup indexes (BARE_INFINITIVE set, modal set) were being built from grammar-feature-toggled filtered wordList rather than the raw wordList. If user's enabled grammar features excluded modal-verb tense data, the rule silently had empty lookup sets → no flags fired. Tight coupling between i18n UI state and core spell-check seam data.
- **Fix:** Decoupled — lookup indexes built from the raw, unfiltered wordList at seam boot. Feature gating happens at explain render time, not at data assembly time.
- **Files modified:** `extension/content/spell-check-core.js` (seam-level index assembly)
- **Verification:** User retested `kan gikk` → flag fires correctly; other rules still respect grammar-feature toggles at render
- **Commit:** `ce343d9`

**3. [Rule 1 - Bug] Modal-verb false-pos on bare infinitive `skrive`**
- **Found during:** Task 3 (smoke test — typed `kan skrive` expecting NO flag, got one)
- **Issue:** The rule was checking if the token following a modal was in the BARE_INFINITIVE set; but `skrive` IS a bare infinitive by definition (infinitive form). The rule's logic was inverted — it should fire when the token is NOT in infinitive form (i.e., a finite verb like `gikk`), not when it's "not in the bare-infinitive set" (which was being populated inconsistently).
- **Fix:** Inverted the check — fires when the following token is a finite verb (in CONJUGATED_FORMS set) and not a modal itself (allowing modal-modal chains like `kan må gå`).
- **Files modified:** `extension/content/spell-rules/nb-modal-verb.js`
- **Verification:** User retested `kan skrive` (no flag) + `kan gikk` (flag) + `kan må gå` (no flag, modal-modal allowed)
- **Commit:** `5aab727`

**4. [Rule 1 - Bug] Dialect-mix CROSS_DIALECT_MAP guard was masked by sisterValidWords check + dot CSS missing**
- **Found during:** Task 3 (smoke test — typed `ikkje` in NB textarea, got no flag in 2 of 3 document states; also first successful flag had no visible red underline)
- **Issue (sub-bug 4a):** The check in nb-dialect-mix.js was ordered `if sisterValidWords.has(token) && map.has(token)`. Tokens that were map-present but data-gap absent from sister-lang validWords silently skipped. Flag intermittency traced to data-gap masking.
- **Issue (sub-bug 4b):** When the rule DID fire, the dot underline wasn't visible in browser. Root cause: `extension/styles/content.css` had `.lh-spell-typo`, `.lh-spell-gender`, etc., but no `.lh-spell-dialect-mix` selector. Rule fired, DOM got the class, but no underline rendered. Node fixture runner doesn't render CSS → blindspot.
- **Fix (4a):** Reordered — CROSS_DIALECT_MAP.has(token) is now the authoritative fire-gate. sisterValidWords is no longer consulted in dialect-mix (only in typo-curated + typo-fuzzy, for their data-gap shield purpose).
- **Fix (4b):** Added `.lh-spell-dialect-mix` underline selector to `extension/styles/content.css` matching existing rule-underline styling but with sky-blue color to visually distinguish dialect-mix from spell-check (red).
- **Fix (4c — systemic guard):** New release gate `scripts/check-rule-css-wiring.js` + paired self-test. Enforces at commit time that every rule_id in `extension/content/spell-rules/*.js` has a matching `.lh-spell-{rule_id}` selector in `content.css`. Plan-to-browser parity closed.
- **Files modified:** `extension/content/spell-rules/nb-dialect-mix.js`, `extension/styles/content.css`, `scripts/check-rule-css-wiring.js` (NEW), `scripts/check-rule-css-wiring.test.js` (NEW), `CLAUDE.md` (release workflow updated)
- **Verification:** User retested — sky-blue underline visible, popover fires on ikkje/ikke across NB/NN/mixed-register paragraphs; release gate self-test PASS
- **Commits:** `a90fcc0` (4a fire-gate), `5718b39` (4b CSS), `c592127` (4c new gate), `54c9137` (CLAUDE.md docs)
