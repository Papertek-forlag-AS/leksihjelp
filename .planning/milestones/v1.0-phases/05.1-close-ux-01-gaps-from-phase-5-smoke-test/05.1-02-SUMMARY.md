## Decisions Made

1. **TYPE_TO_BANK keys `lang`/`nat` not `language`/`nationality`** (Rule 3 - Blocking): The plan prescribed `'language': 'languagesbank'` but `wordIdToBank()` splits the word ID on `_` and looks up the LAST segment. With `_lang`-suffixed IDs, the lookup key must be `lang`. Using the plan's `'language'` key would have silently no-op'd the fast-path and fallen through to the slower `findWordInAllBanks` every request. Fixed inline; added a 3-line comment in the code explaining the suffix-key convention to prevent future re-confusion.

2. **Manifest shape — integer counts, not `{entries: N}` object** (Rule 3 - Blocking): The plan showed `"languagesbank": { "entries": 20 }` but real files use `"nounbank": 2129` (simple integers). Honored the actual shape, not the plan's wording. Otherwise the plan's own verify step (`nb_man.banks.languagesbank.entries`) would have passed while breaking shape-consistency with the 4 existing banks. Adjusted verify probe inline to match (`banks.languagesbank` truthy, without `.entries` accessor).

3. **Search-index files committed alongside data + routing** (plan scope extension): The plan's commit-list included the 7 primary files only. But leaving search-index.json stale against the bank changes would surface as a test-time divergence on any future schema/integrity check. Ran `npm run build:lexicon-search-index` after data landed and included both regenerated index files (NB + NN) in the same atomic commit. Two bytes changed in each (`_metadata.generatedAt` + `_metadata.totalEntries`); zero behavior divergence.

4. **Zero NN linguistic review follow-ups** (plan target acceptance): The plan contemplated that some NN entries might need `_todo: "NN linguistic review"` fallbacks. All 20 NN nationality entries were authored confidently against the grep-verified agent-noun pattern, zero fallbacks needed. If future Språkrådet review surfaces corrections, follow-up can be a trivial data-only PR without bank-structural changes.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] TYPE_TO_BANK keys must match word-ID suffix, not `type` string**
- **Found during:** Task 3 (TYPE_TO_BANK wiring)
- **Issue:** Plan prescribed `'language': 'languagesbank'` and `'nationality': 'nationalitiesbank'`. But `wordIdToBank()` at line 27-38 of `api/vocab/v3/lookup/[language]/[wordId].js` splits the incoming word ID on `_` and looks up the LAST segment in TYPE_TO_BANK. With our IDs `norsk_lang` and `nordmann_nat`, the last segments are `lang` and `nat`. Using the plan's keys would have silently made fast-path routing no-op, forcing every `_lang`/`_nat` lookup to fall through to the slower `findWordInAllBanks` (still works, but defeats the plan's intent).
- **Fix:** Used keys `lang: 'languagesbank'` and `nat: 'nationalitiesbank'`. Added a 3-line inline comment in the code documenting the suffix-key convention.
- **Files modified:** `api/vocab/v3/lookup/[language]/[wordId].js`
- **Verification:** Node-level probe confirmed fast-path routing — `norsk_lang → languagesbank`, `nordmann_nat → nationalitiesbank`, `hund_noun → nounbank` (existing entry, regression check).
- **Committed in:** `bf1bd07b` (atomic Task-3 commit in papertek-vocabulary)
