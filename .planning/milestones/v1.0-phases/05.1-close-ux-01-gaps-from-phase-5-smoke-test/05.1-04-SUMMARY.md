## Decisions Made

1. **Priority 35 (not 30 as CONTEXT.md said).** sarskriving already owns 30; 35 slots dialect-mix between sarskriving and typo-curated (40) so dedupeOverlapping keeps dialect-mix on overlap with typo rules but avoids tying with sarskriving. Documented in rule file header comment + this summary.
2. **CROSS_DIALECT_MAP as the rule-fire gate.** Initial implementation followed the plan verbatim (fire on every sister-valid AND NOT current-valid token). Fixture run produced dialect-mix findings on `Anna`, `kjøkkenet`, `klokka`, `utlandet`, `havet`, `jorda`, `tavla`, `liker`, `kaldt`, `treffer`, `møtte`, `sto` — all genuine Norwegian words that happen to be in one dialect's data but not the other. Restricted to curated map. Net effect: zero false positives across 278 fixture cases; no-fix fallback branch in explain() becomes unreachable in practice but retained for contract shape.
3. **Coexistence with Phase 4 sisterValidWords guards in typo rules.** After removing the Phase 4 guards per the plan's Task 2, the fuzzy rule started flagging `kaldt` as a typo (→`kalde`) because `kaldt` is sister-only (in NB data, not in NN validWords) and not in CROSS_DIALECT_MAP. This regressed `nn-typo-dialect-bare-001`. Restored the sisterValidWords guards in typo-curated + typo-fuzzy with an updated co-existence comment: dialect-mix claims bucket A (curated cross-dialect markers) via dedupeOverlapping; typo rules silently tolerate bucket B (data-gap sister-valid tokens).
4. **NN-side Phase 4 fixtures stay `expected:[]`.** Empirical vocab-seam inspection revealed that NB→NN "dialect" tokens like `jeg`, `ikke`, `bare`, `noen`, `hjem` all appear in NN's validWords via the translation-entry path (each NN entry's `translation` field is seeded as a `type='translation'` wordList entry — so `eg_pron`'s `translation: 'jeg'` puts `jeg` into NN validWords). This means the dialect-mix guard `!validWords.has(t.word)` short-circuits and no finding is emitted. These cases remain valid regression fixtures for the data layer: if the seam's translation-entry behavior ever changes, they would surface the regression.
5. **Lang on finding (Option A from RESEARCH.md #3).** The `(finding) => ({nb, nn})` explain signature is the contract gate's assertion surface and all 5 pre-existing rules use it. dialect-mix emits `lang` on the finding itself and reads `finding.lang` inside explain() to pick direction. No contract-gate changes needed.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 4 → resolved inline] Fire-gate restriction from sisterValidWords\\validWords superset to CROSS_DIALECT_MAP**
- **Found during:** Task 3 (fixture run verification after Task 1+2)
- **Issue:** Plan specified firing on every token in `sisterValidWords AND NOT validWords`. Empirical data-layer reality: that superset is ~6.6K words per direction and includes many morphologically shared forms with data gaps. Firing on the full superset produced dialect-mix findings on 10+ genuine Norwegian words across the clean/codeswitch fixture corpora (Anna, kjøkkenet, klokka, havet, utlandet, jorda, tavla, kaldt, liker, treffer, etc.), regressing nb-clean and nn-clean from 19/19 to 17/19 and 18/19.
- **Fix:** Restrict dialect-mix to fire only when `fixFor(word, lang)` returns a value (i.e., the word is in the lang-appropriate CROSS_DIALECT_MAP). Losing the no-fix fallback branch in practice; retained in explain() for contract shape.
- **Files modified:** extension/content/spell-rules/nb-dialect-mix.js
- **Verification:** nb/clean 19/19, nn/clean 19/19, nb/codeswitch 13/13, nn/codeswitch 13/13 all PASS after refinement.
- **Committed in:** 4a9b4de (refinement committed atomically with Task 1 rule authoring via a late update before first commit — see the check(ctx) comment block in nb-dialect-mix.js).

**2. [Rule 3 - Blocking] Restore sisterValidWords guard in typo-curated + typo-fuzzy (revert Task 2 per-plan change)**
- **Found during:** Task 3 (fixture re-run after Task 2 commit)

- **Fix:** Restore sisterValidWords destructure + early-exit guards in both typo rule files. Update the comment block to describe the new co-existence model (dialect-mix claims bucket A, typo rules silently tolerate bucket B).
- **Files modified:** extension/content/spell-rules/nb-typo-curated.js, extension/content/spell-rules/nb-typo-fuzzy.js

## Issues Encountered

- **Plan assumption vs data-layer reality:** The plan's central assumption (that `sisterValidWords \\ validWords` is a clean "cross-dialect marker" signal) was architecturally invalid under the current data layer. The vocab-seam's buildWordList emits `type='translation'` entries from every bank row's `translation` field, seeding many cross-dialect equivalents into both dialects' validWords sets. Additionally, the sister-dialect's lexicon contains many morphologically shared forms (`kaldt`, `klokka`, `havet`, etc.) that are valid Norwegian in BOTH standards but happen to have data entries in only one. Discovery and resolution required 2 iterations (restrict rule + restore typo-rule guards) but landed at a cleaner architecture than the plan specified.
