## Summary

Phase 05.1 closes four gaps surfaced by the Phase 5 Chrome smoke test (see `.planning/phases/05-student-experience-polish/05-VERIFICATION.md:153-173`). All four gaps are independent: A+B touch the sibling `papertek-vocabulary` repo's data and generator, then flow into leksihjelp via `npm run sync-vocab`; C touches a single spell-rule file + i18n strings; D adds a new spell-rule + inverts Phase 4 dialect-tolerance policy + fixture corpus.

Code-side analysis confirms CONTEXT.md's intent across all four gaps. Discrepancies surfaced below are mostly line-number precision (exact lines to edit) and one substantive issue: **CONTEXT.md states `nb-sarskriving.js` priority is 25; actual code shows priority 30** — the dialect-mix rule at priority 30 would tie, not slot between sarskriving and typo-curated. The honoured CONTEXT.md intent ("win over typo-curated and typo-fuzzy; don't tie with sarskriving") is achievable by picking **priority 35** (between sarskriving 30 and typo-curated 40).

Cross-repo analysis: `papertek-vocabulary` auto-discovers `*bank.json` files in manifest/search-index/list endpoints, so adding new banks is largely additive. The generator defect in Gap A is traced — `scripts/enrich-nb-lexicon.js:444-448` default branch outputs `w + 't'` for every non-special adjective, missing the vowel-ending monosyllabic rule that requires `-tt` doubling (blå → blått, ny → nytt, grå → grått). Fix this branch, not just the lexicon file.

**Primary recommendation:** Plan two cross-repo waves — Wave 1A+1B in `papertek-vocabulary` (fix blå + audit validator; add languages/nationalities banks), Wave 1C+1D in `leksihjelp` (gender copy; dialect-mix rule); Wave 2 in `leksihjelp` runs `npm run sync-vocab` + full release-gate sweep + Chrome smoke test. ### Pattern 1: Rule-File IIFE Shape (for new `nb-dialect-mix.js`)

**What:** Every spell-rule file is a self-contained IIFE that destructures helpers from `host.__lexiSpellCore`, defines a `rule` literal, and pushes to `host.__lexiSpellRules`.
**When to use:** All new spell-check rules.
**Example (verbatim pattern from `nb-modal-verb.js`):**

```javascript
// Source: extension/content/spell-rules/nb-modal-verb.js:9-61
(function () {
  'use strict';
  const host = typeof self !== 'undefined' ? self : globalThis;
  host.__lexiSpellRules = host.__lexiSpellRules || [];
  const { matchCase, escapeHtml } = host.__lexiSpellCore || {};

  const rule = {
    id: '<rule_id>',
    languages: ['nb', 'nn'],
    priority: <N>,
    explain: (finding) => ({
      nb: `…<em>${escapeHtml(finding.original)}</em>…`,
      nn: `…<em>${escapeHtml(finding.original)}</em>…`,
    }),
    check(ctx) {
      const { tokens, vocab, cursorPos, suppressed, lang } = ctx;
      const out = [];
      for (let i = 0; i < tokens.length; i++) {
        const t = tokens[i];
        if (cursorPos != null && cursorPos >= t.start && cursorPos <= t.end + 1) continue;
        if (suppressed && suppressed.has(i)) continue; // Phase 4 ctx.suppressed opt-in
        // ... rule logic ...
        out.push({
          rule_id: '<rule_id>',
          priority: rule.priority,  // Pitfall 1 — priority on every finding
          start: t.start,
          end: t.end,
          original: t.display,
          fix: …,                    // optional; may be null for flag-only findings
          message: `…`,
        });
      }
      return out;
    },
  };

  host.__lexiSpellRules.push(rule);
  if (typeof module !== 'undefined' && module.exports) module.exports = rule;
})();
```

### Pattern 2: Auto-Discovering Bank Iteration (papertek-vocabulary)

**What:** Every `*bank.json` file is picked up by glob in manifest, search-index, and lookup endpoints.
**When to use:** Adding new banks — simply drop the file and it flows through.
**Key evidence:**
- `api/vocab/v3/manifest.js:18` — `.filter(f => f.endsWith('bank.json'))` for per-lang content hash.

- `api/vocab/v3/lookup/[language]/[wordId].js:41-47` — `findWordInAllBanks` iterates every `*bank.json` as a fallback.
- Per-lang `manifest.json` (e.g., `lexicon/nb/manifest.json`) has a `"banks": {…}` count block — update manually or via a regen script.

### Pattern 3: i18n Key Lookup (Gap C helper design)

**What:** `extension/i18n/strings.js` currently exposes `self.__lexiI18n.t(key)` keyed by the CURRENT UI locale. Gap C needs **per-target-locale** lookup (the `explain` callable produces both `nb` and `nn` strings in one call, routed by document lang, not UI lang).
**When to use:** Any spell-rule whose explain copy needs locale-specific vocabulary (gender labels, dialect names).
**Recommended helper shape (Claude's Discretion — planner picks placement):**

```javascript
// Option A — add to __lexiSpellCore:
// spell-check-core.js exposes getString(key, lang) that reads __lexiI18n.STRINGS[lang][key] directly.
function getString(key, lang) {
  const STRINGS = (typeof self !== 'undefined' && self.__lexiI18n && self.__lexiI18n.STRINGS) || {};
  return (STRINGS[lang] && STRINGS[lang][key]) || (STRINGS.nb && STRINGS.nb[key]) || key;
}

// Option B — rule-local helper inlined in nb-gender.js:
// Only one rule (for now) needs cross-locale lookup, so inline is fine.
```

**Research verdict:** Option A scales better (Gap D's nb-dialect-mix will also need `getString('dialect_nb', 'nb')` / `'dialect_nn'`-style lookups if dialect names are externalized — CONTEXT.md says "status quo" = dialect-mix keeps copy in rule file, so only Gap C strictly needs the helper). Plan should align placement with whether future rules are expected to reuse.

### Anti-Patterns to Avoid

- **Embedding gender labels directly in `nb-gender.js`:** CONTEXT.md locks the i18n-strings source. Direct embedding would duplicate labels across NB and NN callables and make future locale additions (English, deferred) a rule-logic edit instead of a strings-file edit.
- **Reading `__lexiI18n.t(key)` from the rule file:** `t()` routes via `_uiLang`, the Chrome-storage UI locale — NOT the document's active register. Gap C needs to resolve labels in both NB and NN in a single `explain()` call. Use direct `__lexiI18n.STRINGS[lang][key]` access instead.
- **Adding `sisterValidWords` to a Gap D new rule and ALSO keeping early-exit in typo-curated/typo-fuzzy:** The whole point of Gap D is to flip the flag/no-flag decision. Leaving both would no-op the new rule for the exact cases it's supposed to catch.
- **Using `type: 'noun'` for language names in `languagesbank.json`:** Would pollute `nounGenus` Map via vocab-seam-core.js:473-477 and could introduce spurious gender-rule flags. Use a distinctive `type` (e.g., `'language'` / `'nationality'`) — see Open Questions.
- **Forgetting `priority` on emitted findings:** renderExplain uses `(rule_id, priority, lang)` three-way lookup at `spell-check.js:473-480` — findings without `priority` fall through to the less-specific match and may hit the wrong rule's explain callable when two rules share a `rule_id` (e.g., typo-curated and typo-fuzzy both emit `rule_id: 'typo'`).

## Common Pitfalls

### Pitfall 1: CONTEXT.md priority-number discrepancy for sarskriving

**What goes wrong:** CONTEXT.md §Gap D says `nb-sarskriving.js` has priority 25, and the new `nb-dialect-mix.js` at priority 30 "slots between sarskriving (25) and typo-curated (40)." In code, `nb-sarskriving.js:45` is priority **30**, not 25. // NEW FILE — shape inspired by existing scripts/audit-nn-adjectivebank.js
```javascript
import { readFileSync } from 'fs';
import { join } from 'path';

const BANKS = [
  ['nb', 'vocabulary/lexicon/nb/adjectivebank.json'],
  ['nn', 'vocabulary/lexicon/nn/adjectivebank.json'],
];

// Known irregulars — whitelist. Planner to expand as audit surfaces cases.
const IRREGULAR = new Set([
  'liten_adj', 'god_adj', 'gammal_adj', 'gamal_adj', 'dårlig_adj',  // ...
]);

// Expected monosyllabic-vowel roots where neuter is -tt
const MONOSYLLABIC_VOWEL = /^[bcdfghjklmnpqrstvwxz]?[aeiouyæøå]$/;

function predictNeuter(root) {
  if (root.endsWith('e')) return root;                            // same as base
  if (root.endsWith('ig')) return root + 't';
  if (root.endsWith('sk')) return root;
  if (root.endsWith('t')) return root;
  if (/(dd|tt|nn|mm|ll|ss|kk|pp)$/.test(root)) return root;        // double consonant
  if (MONOSYLLABIC_VOWEL.test(root)) return root + 'tt';           // blå→blått
  return root + 't';                                               // default
}

let defects = 0;
for (const [lang, path] of BANKS) {
  const bank = JSON.parse(readFileSync(path, 'utf8'));
  for (const [id, entry] of Object.entries(bank)) {
    if (id.startsWith('_')) continue;
    if (IRREGULAR.has(id)) continue;
    const positiv = entry.declension?.positiv;
    if (!positiv) { console.warn(`[${lang}] ${id}: missing declension.positiv`); defects++; continue; }
    const expected = predictNeuter(entry.word.toLowerCase());
    if (positiv.noytrum !== expected) {
      console.warn(`[${lang}] ${id}: noytrum="${positiv.noytrum}" expected="${expected}"`);
      defects++;
    }
  }
}
process.exit(defects > 0 ? 1 : 0);
```

```javascript
 * Spell-check rule: dialect-mix detection (priority 35).
 *
 * Phase 05.1 / UX-01 gap closure. NB and NN are two distinct official written
 * standards of Norwegian; cross-standard tokens are student errors, not dialect
 * choice. Flags tokens that are valid in the sister dialect's validWords but
 * NOT in the current dialect's validWords, emitting a curated cross-dialect
 * fix when known.
 *
 * Reverses Phase 4 / SC-03 dialect-tolerance. Co-fires with typo-curated and
 * typo-fuzzy; `dedupeOverlapping` keeps the lower-priority finding.
 * Priority 35 slots between sarskriving (30) and typo-curated (40).

  'use strict';
  const host = typeof self !== 'undefined' ? self : globalThis;
  host.__lexiSpellRules = host.__lexiSpellRules || [];
  const { escapeHtml } = host.__lexiSpellCore || {};

  // Two directional maps — NB doc session flags NN-token, NN doc session flags NB-token
  const NN_TO_NB = new Map([
    ['ikkje', 'ikke'], ['eg', 'jeg'], ['berre', 'bare'], ['nokon', 'noen'],
    ['kva', 'hva'], ['kvar', 'hvor'], ['korleis', 'hvordan'], ['kven', 'hvem'],
    ['kvifor', 'hvorfor'], ['mykje', 'mye'], ['ho', 'hun'], ['dei', 'de'],
    ['heim', 'hjem'], ['vere', 'være'], ['meinte', 'mente'], ['høyrer', 'hører'],
    // … planner expands to ~30-40 entries
  ]);
  const NB_TO_NN = new Map([
    ['ikke', 'ikkje'], ['jeg', 'eg'], ['bare', 'berre'], ['noen', 'nokon'],
    ['hva', 'kva'], ['hvor', 'kvar'], ['hvordan', 'korleis'], ['hvem', 'kven'],
    ['hvorfor', 'kvifor'], ['mye', 'mykje'], ['hun', 'ho'], ['de', 'dei'],
    ['hjem', 'heim'], ['være', 'vere'], ['mente', 'meinte'], ['hører', 'høyrer'],
    // … planner expands to ~30-40 entries
  ]);

  const rule = {
    id: 'dialect-mix',
    languages: ['nb', 'nn'],
    priority: 35,
    explain: (finding) => {
      const hasFix = !!finding.fix;
      if (finding.lang === 'nb') {
        // NB doc → student typed NN → teach with fix, or hedge fallback
        return hasFix
          ? {
              nb: `<em>${escapeHtml(finding.original)}</em> er nynorsk — prøv <em>${escapeHtml(finding.fix)}</em>.`,
              nn: `<em>${escapeHtml(finding.original)}</em> er nynorsk — prøv <em>${escapeHtml(finding.fix)}</em>.`, // same copy

              nb: `<em>${escapeHtml(finding.original)}</em> er nynorsk. Skriv du bokmål?`,
              nn: `<em>${escapeHtml(finding.original)}</em> er nynorsk. Skriv du bokmål?`,
            };
      }
      // NN doc → student typed NB
      return hasFix
        ? {
            nb: `<em>${escapeHtml(finding.original)}</em> er bokmål — prøv <em>${escapeHtml(finding.fix)}</em>.`,
            nn: `<em>${escapeHtml(finding.original)}</em> er bokmål — prøv <em>${escapeHtml(finding.fix)}</em>.`,
          }
        : {
            nb: `<em>${escapeHtml(finding.original)}</em> er bokmål. Skriv du nynorsk?`,
            nn: `<em>${escapeHtml(finding.original)}</em> er bokmål. Skriv du nynorsk?`,
          };
    },

      const { tokens, vocab, cursorPos, suppressed, lang } = ctx;
      const validWords = vocab.validWords || new Set();
      const sisterValidWords = vocab.sisterValidWords || new Set();
      const fixMap = lang === 'nb' ? NN_TO_NB : NB_TO_NN;
      const out = [];
      for (let i = 0; i < tokens.length; i++) {

        });
      }
      return out;
    },
  };
```

## Open Questions

1. **Gap B: bank type field vs existing `nounbank` / `adjectivebank` fit**
   - What we know: Norwegian `norsk`/`engelsk`/`tysk` etc. are primarily adjective forms (substantivized for language reference); `nordmann`/`svenske` are clean nouns; `dansk` is adjective. CONTEXT.md locks new banks.
   - What's unclear: Does every entry in `languagesbank.json` carry `type: "language"`, or do entries carry their `type: "adj"`/`"noun"` + bank=`"languagesbank"` is the only distinguishing field? The choice affects whether `nounGenus` picks up genus data (seam condition at vocab-seam-core.js:473 is `entry.bank === 'nounbank' || entry.type === 'nounform' || entry.type === 'plural'`).
   - Recommendation: Use `type: "language"` and `type: "nationality"` as first-class types. Adjectivebank entries keep `type: "adj"`; these don't. Planner decision during Gap B schema design.

2. **Gap A: how extensive is the NN adjectivebank audit?**
   - What we know: Gap A applies "the same audit to NN in parallel." But NN adj generator is likely a transformation of NB (per `_generatedFrom: nb (transformed)` in `lexicon/nn/manifest.json:14`).
   - What's unclear: If NN derives from NB and NB gets the `-tt` fix, does the NN re-derivation pick it up automatically? Or is there an independent NN generator with its own monosyllabic-vowel bug?

3. **Gap D: `explain()` signature — `(finding)` vs `(finding, lang)`**
   - What we know: All 5 existing popover-surfacing rules use `explain: (finding) => ({nb, nn})`. `spell-check.js:491` calls `rule.explain(finding)` (single arg).
   - What's unclear: Gap D's dialect-mix rule needs lang context to pick the right copy ("er nynorsk" vs "er bokmål"). Options: (a) emit `lang` on every finding, (b) extend the contract to `(finding, lang)`.
