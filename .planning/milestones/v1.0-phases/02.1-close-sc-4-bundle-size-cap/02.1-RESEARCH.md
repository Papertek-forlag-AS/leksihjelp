## Architecture Patterns

### Pattern: Hard Numeric Threshold Release Gate (locked in by Plan 02-04)

Per STATE.md decision-log: *"Release-gate pattern locked in — scripts named `scripts/check-*.js` run with zero args, produce a diagnostic + PASS/FAIL line, exit 0/1 against a hard numeric threshold, and register in CLAUDE.md's Release Workflow as must-exit-0 steps."*

This phase **preserves** the pattern. Only the threshold number moves. Architecture invariants:
- Script signature: zero CLI args, runs `npm run package` first, measures `backend/public/lexi-extension.zip` via `fs.statSync`, parses `unzip -l` for per-directory breakdown.
- Output contract: a "PASS" or "FAIL" line, a comma-separated byte count, a per-directory breakdown table.
- Exit codes: `0` if `zipSize <= CEILING_BYTES`, `1` otherwise (or if packaging itself failed).
- Registered as Release Workflow step 2 in `CLAUDE.md`.

### Pattern: Source-Tree Invariant — `extension/data/*.json` stays pretty-printed

CLAUDE.md line 358: *"never manually minify `extension/data/*.json` in the source tree — keep the repo copies pretty-printed for contributor readability."* The minification happens inside the staging dir during `npm run package`. This phase does **not** touch the data files or the staging logic — it inherits the invariant unchanged.

### Anti-Patterns to Avoid

- **Silently editing `CEILING_BYTES` without updating the prose.** Plan 02-04 SUMMARY (line 155) and 02-04 PLAN (line 206) explicitly forbade this in the previous round. The fix here is to edit the constant **AND** rewrite the prose so the new number is reflected accurately throughout — they must move together.
- **Rewriting historical archive files.** `.planning/phases/02-data-layer-frequency-bigrams-typo-bank/` documents ship as the audit trail of what was decided when the 10 MiB number was alive. Editing them retrofits a different past. Leave untouched.
- **Touching `STATE.md` historical log entries.** Lines 103 and 127 of STATE.md repeat the "publicly-stated 10 MB promise" phrasing — these are historical decision-log entries documenting the Phase 02-04/02-05 reasoning at the time. Treat as historical record per CONTEXT's Docs correction principle. (The Roadmap Evolution section + the resolved Blocker entry should be **updated** to reflect Phase 02.1's resolution, but the historical decision rows stay.)

## Common Pitfalls

### Pitfall 1: Updating only the constant, not the prose
**What goes wrong:** `CEILING_BYTES` becomes `20 * 1024 * 1024` but the script's header comment still says "10 MiB ceiling" and CLAUDE.md's Release Workflow still describes a "10 MiB ceiling". Future readers see contradictory numbers and don't know which is authoritative.
**Why it happens:** The constant is what the test asserts and what the gate enforces; the prose feels secondary. But the prose is what humans read first.
**How to avoid:** Treat the constant edit and the prose-rewrite passes as one atomic unit of work. The plan should bundle them.
**Warning signs:** A grep for `10\s?MiB|10 MB|10485760` after the edits returns hits anywhere outside the historical archive (`.planning/phases/02-data-layer-frequency-bigrams-typo-bank/`) or the documenting-history rows of STATE.md.

### Pitfall 2: Forgetting the test file's constant + test name
**What goes wrong:** `scripts/check-bundle-size.js` ships with `CEILING_BYTES = 20 * 1024 * 1024`, but `scripts/check-bundle-size.test.js:27` keeps `10 * 1024 * 1024`. The test still passes against the current zip (10.11 MiB is under both old and new caps), so it appears green. But the test's *intent* — assert that the gate matches the cap — is silently broken.
**Why it happens:** CONTEXT.md correctly flags Test 4/5 (line 124's test name string) but the constant at line 27 needs the same edit and is easy to miss.
**How to avoid:** When editing `scripts/check-bundle-size.js`, also edit `scripts/check-bundle-size.test.js` in the same diff. Re-run `node scripts/check-bundle-size.test.js` and confirm all behavior tests still pass.
**Warning signs:** `grep -n CEILING_BYTES scripts/` returns mismatched values across the two files.

### Pitfall 3: Rewriting historical archive files
**What goes wrong:** Editor sweeps `.planning/phases/02-data-layer-frequency-bigrams-typo-bank/02-04-SUMMARY.md` and 02-05-SUMMARY.md to "fix" the 10 MiB references. Suddenly the audit trail says Phase 02-04 succeeded against a 20 MiB cap, which it didn't.
**Why it happens:** The archive files contain the same prose patterns as the live docs, and a global find-and-replace is faster than a manual sweep.
**How to avoid:** Plan must explicitly list the **5 live files** to edit and explicitly exclude `.planning/phases/02-data-layer-frequency-bigrams-typo-bank/`. No global replace.
**Warning signs:** Any `git diff` line touching the 02-data-layer-frequency-bigrams-typo-bank/ directory.

### Pitfall 4: Updating STATE.md historical entries instead of the live blocker
**What goes wrong:** STATE.md lines 103 and 127 say "publicly-stated 10 MB promise" — these are historical decision-log entries documenting what was thought during Plan 02-04 and Plan 02-05. Editing them rewrites history. The correct STATE.md edit is to **resolve** the live "Phase 2 bundle-size 10 MiB ceiling contingency — SC-4 STILL OPEN" blocker (also at line 127, in the Blockers/Concerns section) by marking it closed and pointing to Phase 02.1.
**Why it happens:** STATE.md mixes historical log + current state. The two need different treatment.
**How to avoid:** STATE.md gets ONE addition (resolve the SC-4 blocker, mark Phase 02.1 done) and ZERO edits to the historical decision rows under "## Accumulated Context > Decisions". The /gsd:resume-work or /gsd:verify-work tooling typically handles this.
**Warning signs:** Diffs that touch lines 103 or the historical decision log entries.

### Pitfall 5: SC-4 rewording diverges between ROADMAP and REQUIREMENTS
**What goes wrong:** ROADMAP.md says "stays under 20 MiB" but REQUIREMENTS.md DATA-03 still says "~10 MB ceiling". Future phases reference one or the other and inherit confusion.
**Why it happens:** Two files, two passes, easy to forget the second.
**How to avoid:** Bundle ROADMAP.md and REQUIREMENTS.md edits in the same task. The phrasing doesn't have to be identical word-for-word, but the **number** must match.
**Warning signs:** A grep for `10\s?MiB|10 MB` in the live (non-archive) docs returns hits after the phase commit.

### Pitfall 6: Re-litigating remediation choices
**What goes wrong:** The plan's executor sees the audit work in `02-05-AUDIT.md` and starts proposing English-removal or audio-strip refactors as "while we're here, let's do this properly."
**Why it happens:** The audit evidence is detailed and tempting. The remediation decision feels under-baked because it's "just bumping a number."
**How to avoid:** CONTEXT.md is explicit — those are deferred. The 20 MiB number is set; the existing zip is already under it. No further work in this phase.
**Warning signs:** Any task that mentions BUNDLED_LANGUAGES, audio/de/, en.json, or sync-vocab.js.
