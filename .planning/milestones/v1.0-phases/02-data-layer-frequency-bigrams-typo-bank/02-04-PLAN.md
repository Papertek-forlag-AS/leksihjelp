<objective>
Close Phase 2 with (a) a release-gate script that enforces the 10 MiB extension zip ceiling (ROADMAP success criterion #4), (b) JSON minification at package time as the first-line size-reduction strategy, and (c) a CLAUDE.md release-workflow update that wires the gate permanently into the release ritual.

**Two-outcome reality (addresses checker scope_sanity warning):** The pre-Phase-2 zip was already 10.26 MiB — over the 10 MiB ceiling by ~260 KB BEFORE any Phase 2 data lands. Plans 02-01 / 02-02 / 02-03 add net ~400-800 KB gzipped of new data (freq tables, regrown bigrams, expanded typos). JSON minification of `extension/data/*.json` at package time is the first-line remediation (research: pretty-printed JSON is 15-25% wasteful; minification saves ~1-2 MB inside the zip because the 35 MB uncompressed lexicon compresses better when whitespace is already stripped). Whether minification ALONE brings the bundle under 10 MiB is an empirical question that this plan answers when executed — there are two legitimate outcomes:

**Outcome A — Minification succeeds.** After all Phase 2 data additions and JSON minification, the packaged zip measures under 10 MiB. `npm run check-bundle-size` exits 0. Phase 2 success criterion #4 SATISFIED. Ship Phase 2 as planned. The bundle-size gate ships as a permanent guard against future regression.

**Outcome B — Minification is insufficient.** The zip is still over 10 MiB after all three Phase 2 data plans land and minification is applied. This plan STILL SHIPS — the bundle-size script, minification pipeline, and CLAUDE.md release-workflow update all land as permanent infrastructure. What does NOT ship in Phase 2 is a zip that passes the ceiling: `npm run check-bundle-size` exits 1 with a clear FAIL + per-directory breakdown. Phase 2 closes with **success criterion #4 explicitly documented as a Blocker** in STATE.md (the existing Phase-2.1 contingency entry gets updated with the exact post-minification byte count and directory breakdown). **Phase 2.1** (per ROADMAP's "decimal phases for urgent insertions" convention) opens to make the product decision that exceeds this plan's scope: strip bundled `audio/de/` and fetch on first use (saves ~3 MB inside zip; breaks offline German TTS), trim rarely-used vocab entries (breaks offline pledge), split the lexicon into core + optional chunks loaded on demand, or explicitly bump the publicly-stated 10 MB ceiling (requires user sign-off and a landing-page update). Those are product decisions, not research calls, and out of scope for Phase 2.

This plan does NOT pre-commit to either outcome. It ships the tooling (which is useful in BOTH outcomes) and records the result honestly. If outcome B materialises, the fail-loud script + STATE.md update is the responsible escalation; silently shipping a 10.6 MiB zip because minification almost-worked would violate the publicly-stated 10 MB ceiling.

Purpose:
- Make success criterion #4 measurable and automated: "Total packaged extension zip size from `npm run package` stays within the 10 MB ceiling, verified in the release checklist."
- Prevent silent bundle-size creep from future vocab/data growth.
- Apply the cheapest size-reduction (JSON minification) that is invisible to consumers.
- If that's not enough, surface the gap clearly for a Phase 2.1 product decision rather than hide it.

This plan MUST run after the three Wave-1 plans are complete because it needs the combined output of all three data additions to measure the realistic packaged size. Waves 1 and 2 cannot parallelise on this plan — the measurement needs their output to be the final word.

6. `process.exit(size <= CEILING_BYTES ? 0 : 1);`
