<objective>
Ship a Node build script that ingests the NB N-gram 2021 book-corpus unigrams (CC-0, from Nasjonalbiblioteket Språkbanken) and emits two sidecar Zipf frequency files under `extension/data/` — one for NB, one for NN — each well under 200 KB gzipped. The script streams the gzipped CSV (1.04 GB, far too big for in-memory load), filters by the corpus `lang` column (ISO 639-3 `nob` / `nno`, NOT internal `nb` / `nn`), intersects against `validWords` from Phase 1's locked `buildIndexes()` seam so only shipped words have frequency entries, converts raw counts to Zipf floats (`zipf = log10(freq / totalTokens) + 9`, rounded to 2 decimals), drops the rare tail below a Zipf floor (default 3.0 ≈ one per million), and writes compact JSON. The same build script produces both language files from the same single download (one corpus, two language passes).

Purpose: DATA-01 is the single highest-leverage ranking-quality data investment (Phase 3 SC-01 cannot fix the `berde → berre` vs `bedre` ranking bug without unigram frequency). The seam already exposes `getFrequency(word)` returning a Map<string, number> that is currently always empty; Phase 2 DATA-01 populates the Maps without changing the seam signature. Phase 3 consumes via the seam without caring about where the data came from.

Output:
- `scripts/build-frequencies.js` (new Node CommonJS script, ~150-200 LOC, zero npm dependencies).
- `extension/data/freq-nb.json` and `extension/data/freq-nn.json` (new sidecar tables).
- `package.json` gains a `"build-frequencies": "node scripts/build-frequencies.js"` script.
- `.gitignore` gains a `corpus/` exclusion so the 1 GB+ cache never reaches git.
- The regression fixture (`npm run check-fixtures`) continues to exit 0 — DATA-01 adds sidecar data only, no rule behavior changes.
