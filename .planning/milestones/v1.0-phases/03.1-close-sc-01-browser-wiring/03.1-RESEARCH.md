## Summary

The v1.0 audit surfaced an integration defect in the browser-side wiring of the Zipf tiebreaker. The math is correct, the data is present on disk (`extension/data/freq-{nb,nn}.json`, ~200 KB + ~173 KB), the seam hydrates a `Map` inside its internal `state.freq`, and the fuzzy rule reads `vocab.freq.get(cand)` behind a null guard. The break is a single adapter: `extension/content/spell-check.js:207-213` assembles the `vocab` object handed to `CORE.check()` from five getters (`getNounGenus`, `getVerbInfinitive`, `getValidWords`, `getTypoFix`, `getCompoundNouns`) and omits the frequency Map entirely. Because the fuzzy rule's `scoreCandidate` is defensively coded (`if (vocab && vocab.freq) { ... }`), the omission is silent — candidates score without the Zipf term and ranking reverts to the pre-Phase-3 formula. The fixture runner (`scripts/check-fixtures.js:138`) passes the *entire* `buildIndexes()` return object — which does include `freq` — so the Zipf path is exercised on Node but not in Chrome.

The fix is trivially small (one new getter on the seam, one additional line on the adapter's vocab assembly) but must ship with a contract test that prevents the defect from reappearing. The current `03-VERIFICATION.md` marked the `nb-typo-fuzzy.js → vocab.freq` link as "WIRED" by grepping the rule file itself — a verification method that proved the rule reads `vocab.freq` but not that its caller actually supplies it. The new verification should walk the full call chain end-to-end: seam getter -> adapter object -> core.check -> rule.check -> scoreCandidate -> Zipf term applied.

**Primary recommendation:** Add `VOCAB.getFreq()` returning the `state.freq` Map reference (matching the `getBigrams()` precedent in the same file), then include `freq: VOCAB.getFreq()` in the runCheck vocab object. Ship with a new fixture-runner-adjacent assertion or a focused Node-side contract test that fails loud if the adapter ever ships a vocab object without a populated `freq` on NB/NN. Preserve the 140/140 fixture pass rate. Do not modify the Zipf math, the multiplier, or the rule's null-guard.

### Pattern 1: Seam getter returning Map reference

**What:** Add `getFreq()` to `__lexiVocab` that returns `state.freq` if populated, else a fresh empty `Map`. This is a direct analog of the existing `getBigrams()` method in the same file.

**When to use:** Whenever a content-script adapter needs to pass a seam-owned index into a pure consumer. The precedent for reference-passing (vs. per-key lookups) is already in place for `getBigrams()` and for the Set/Map getters `getValidWords()`, `getNounGenus()`, etc.

**Example:**

```javascript
// extension/content/vocab-seam.js — ADD this method
// Alongside getBigrams() around line 230:

// Frequency Map (Zipf-scored unigrams from freq-{lang}.json sidecar).
// NB/NN: populated ~13k / ~11k entries. DE/ES/FR/EN: empty Map (no sidecar shipped).
// Consumers (spell-rules/nb-typo-fuzzy.js) pass this through core.check -> rule.check
// and read via vocab.freq.get(word). Empty Map is returned rather than null so
// consumers can null-guard on `vocab && vocab.freq` AND call .get() safely.
getFreq: () => (state && state.freq instanceof Map) ? state.freq : new Map(),
```

Pattern rationale: **same shape as `getBigrams()`** (which returns the normalized bigrams object or null) but with an empty-Map-default instead of null-default. The Map-default matches the convention used for `getValidWords()` / `getNounGenus()` — callers don't need to null-check, they just call `.get()` / `.has()`.

### Pattern 2: Adapter-side vocab object assembly

**What:** `spell-check.js:runCheck` constructs the vocab object passed to `CORE.check`. Every time a new piece of seam state needs to reach a rule, this assembly list must grow — this is the coupling that just failed us.

**When to use:** The current hand-assembly pattern is how the code is structured. Three options for Phase 03.1:

1. **Minimal additive fix (RECOMMENDED):** Add `freq: VOCAB.getFreq()` to the existing object literal. Four-character-plus diff. Matches existing pattern exactly.

2. **Refactor into a single getter on the seam:** `VOCAB.getSpellCheckVocab()` returning the full object. Reduces future drift risk but touches both files and breaks pattern parity (the other getters stay per-field).

3. **Spread a canonical "vocab-for-rules" object:** Add `getAllIndexes()` returning a frozen snapshot; adapter does `const vocab = VOCAB.getAllIndexes()`. Cleanest but largest surface-area change.

Recommendation: **Option 1** (additive), plus a contract assertion (see Pattern 3) that guards against future adapter drift without imposing the wider refactor.

**Example:**

```javascript
// extension/content/spell-check.js:207 — ADD freq field
const vocab = {
  nounGenus:      VOCAB.getNounGenus(),
  verbInfinitive: VOCAB.getVerbInfinitive(),
  validWords:     VOCAB.getValidWords(),
  typoFix:        VOCAB.getTypoFix(),
  compoundNouns:  VOCAB.getCompoundNouns(),
  freq:           VOCAB.getFreq(),   // Phase 03.1: bridge Zipf tiebreaker into browser runtime (SC-01)
};
```

### Pattern 3: Browser-adapter contract assertion (new gate)

**What:** Add a Node-side contract test that re-creates the `runCheck` vocab object shape and asserts all fields a rule might read. The current verification-by-grep (`grep 'vocab.freq' ...`) verified the rule *reads* the field; it did not verify the adapter *supplies* it.

**When to use:** Any time the adapter pattern is load-bearing for a requirement. The existing release-gate pattern locked in Phase 2 (`check-*.js` + paired `check-*.test.js`) gives a natural home for this.

**Example structure:**

```javascript
// scripts/check-spell-check-vocab-contract.js (NEW, proposed)
// OR: inline assertion block added to scripts/check-fixtures.js
//
// Purpose: parse extension/content/spell-check.js, extract the vocab object
// literal inside runCheck, and assert it includes every field that any
// registered rule's scoreCandidate/check body reads via vocab.X.
//
// Minimum viable check: static substring check on spell-check.js for
// "freq: VOCAB.getFreq()" OR ".freq" in the runCheck vocab literal.
// Exits 1 with file:line output if the defect class returns.
```

Simpler alternative: add one line to `scripts/check-fixtures.js` after loading vocab that also reads the adapter file and asserts the literal contains `freq:`:

```javascript
// scripts/check-fixtures.js — proposed inline addition near top of main()

  path.join(__dirname, '..', 'extension', 'content', 'spell-check.js'),
```

This rides on the existing fixture-runner release gate instead of adding a new one.

### Anti-Patterns to Avoid

- **Verifying a link by grep'ing only one side of it:** The 03-VERIFICATION.md row "`nb-typo-fuzzy.js → vocab.freq Map` WIRED" was wrong because it grep'd the *rule* file for the read, not the *adapter* file for the write. End-to-end link verification must check both endpoints or walk the call chain.
- **Reaching into `state` directly from the adapter:** `self.__lexiVocab` owns `state`; exposing it breaks encapsulation and makes future seam refactors fragile.
- **Adding a per-word `VOCAB.getFrequency(word)` call inside `scoreCandidate`:** Plausible-looking shortcut that (a) couples the rule to the seam surface (violates INFRA-03/INFRA-04), (b) does the Map lookup via a method call 2-3× slower than a direct `Map.get()` over a large candidate list.
- **Adding the Zipf term without the contract assertion:** The defect's failure mode is silent; any future defensive coding that drops `freq` (e.g., refactoring the adapter to filter fields for safety) would silently break SC-01 again.

**Key insight:** The defect is small, well-located, and sits in a code path where the fixture runner and the browser adapter diverge *on purpose* (Node runner uses `buildIndexes()` output directly for ergonomics; browser adapter goes through the seam's public surface to preserve encapsulation). The right fix restores parity at the seam-surface layer (adding `getFreq()` to match the existing `getBigrams()` pattern) rather than changing the divergence strategy.

## Common Pitfalls

### Pitfall 1: Empty-Map-default vs null-default in the new getter

**What goes wrong:** Pattern-match `getBigrams()` (returns null when missing) and write `getFreq: () => state?.freq ?? null`. The rule's null-guard `if (vocab && vocab.freq)` then behaves identically to today (skips Zipf). Fixture runner passes because it bypasses the adapter entirely.

**Why it happens:** Two precedents exist in the seam — `getBigrams()` (null-default) and `getValidWords()` (empty-Set-default). Picking the wrong one silently matters.

**How to avoid:**
- Return `new Map()` when state is not ready yet, and return `state.freq` (the Map reference) otherwise. Match `getValidWords()` / `getNounGenus()` style, not `getBigrams()` style.
- The rule's null-guard is `if (vocab && vocab.freq)` — an empty Map is truthy, so the guard passes and `vocab.freq.get(cand)` returns `undefined`. `if (typeof z === 'number')` below it catches the undefined. No Zipf boost, no crash. Correct behavior.
- Fail-loud alternative: throw from the getter if `state?.freq` isn't a Map once ready. Rejected for robustness — content scripts shouldn't throw during startup races.

**Warning signs:** Contract assertion passes (field is present) but fixture file `nb-typo-zipf-001/002` flips from pass to fail (no Zipf boost applied) after the wiring change lands.

### Pitfall 2: Relying on the existing fixture suite to catch browser-runtime regressions

**What goes wrong:** Land the wiring fix, run `npm run check-fixtures` (140/140 pass), mark SC-01 closed. But the fixture suite continues to use `buildIndexes()` output directly — it would ALSO pass if the adapter never read `freq` at all. The Node-side P/R/F1 is unchanged.

**Why it happens:** The fixture runner is a white-box test of the core; the browser adapter is the integration layer, and fixtures don't exercise it. That's precisely why the defect survived Phase 3 verification.

**How to avoid:**
- Add a Node-side contract assertion (described in Pattern 3) that proves the adapter literal contains `freq: VOCAB.getFreq()`. This is browser-side coverage the fixture runner cannot otherwise provide.
- OR: introduce a micro-harness that instantiates a fake `__lexiVocab` surface, imports `spell-check.js`, and calls `window.__lexiSpell.recheck()` while asserting the vocab handed to `CORE.check` includes `freq`. More work, more realistic. Deferred unless the contract assertion is deemed insufficient.
- Document in SUMMARY.md that the fixture suite DOES NOT exercise the adapter path — a follow-up readme/CLAUDE.md note helps future audits avoid the same trap.

**Warning signs:** Fixture suite green, but typing a zipf-target typo in Chrome still picks the wrong suggestion (e.g., `hagde` picks `hadde`'s d=1 competitors based on length rather than Zipf).

### Pitfall 3: Cursor-position skip collides with the zipf fixture in interactive browser

**What goes wrong:** The fuzzy rule (`nb-typo-fuzzy.js:111`) skips tokens whose span straddles the cursor. Fixture runner passes `cursorPos: undefined` implicitly -> skip never fires -> every fixture case runs. In browser, the user's cursor is almost always at the end of the word they just typed -> every zipf-target case is skipped until they type a space or move the cursor.

**Why it happens:** This is already correct behavior for interactive editing (we don't want to flag mid-thought), but if a human smoke-tester types `hagde` and looks at the popup without moving their cursor past the token, no flag appears. They might conclude "fix doesn't work."

**How to avoid:**
- Smoke-test pattern: type the target word THEN press space or arrow-right before inspecting. Documented in Phase 1 verification playbook.
- Alternative: open Chrome DevTools and manually call `window.__lexiSpell.recheck()` after clicking somewhere else (cursor left the token).
- Mark this explicitly in SUMMARY.md's "how to smoke-test the fix" section.

**Warning signs:** "It doesn't work!" from a user who didn't move the cursor. Reproducer: type + space + wait 800ms (debounce) + inspect dot/popover.

### Pitfall 4: Not normalizing lookup case in the Map

**What goes wrong:** `state.freq` is already lowercase-keyed (`vocab-seam-core.js:514-519`) but the fuzzy rule passes `cand` (already lowercased because validWords is a `Set<string>` of lowercase forms). If a future refactor changes the iteration to use `entry.display` or some mixed-case source, Zipf lookups miss.

**Why it happens:** Map key casing is an implicit contract. The seam lowercases at build time; the rule lowercases at tokenize time; both agree today.

**How to avoid:**
- Keep the current design. Do not add any `.toLowerCase()` calls inside `scoreCandidate` — that's a signal the invariant just drifted.
- A defensive `vocab.freq.get(cand.toLowerCase())` is safe but masks the real contract. Leave it as `vocab.freq.get(cand)` and rely on the invariant.

**Warning signs:** Some NB candidates get Zipf boosts, some don't. Root cause: a candidate somehow entered `validWords` with mixed case. Check seam emission, not the rule.

### Pitfall 5: Bundle-size regression from introducing the getter (false alarm)

**What goes wrong:** "Adding `getFreq()` might push the packaged zip past the 20 MiB cap." Not true. The `freq-{nb,nn}.json` sidecars are already bundled (376,685 bytes uncompressed, pretty-printed in-tree; minified to much less by `npm run package`). They're served via `chrome.runtime.getURL` already (confirmed in `vocab-seam.js:113-122` `loadRawFrequency` — shipped since Plan 03-01). Adding a method to the seam adds ~1-2 lines of source, which is well under the per-file minifier's noise floor. Current zip is 10.12 MiB / 20 MiB cap (9.88 MiB headroom).

**Why it happens:** Mis-remembering the Phase 02.1 sizing discussion.

**How to avoid:** `npm run check-bundle-size` as part of the release-gate cadence. Exit 0 required.

**Warning signs:** None expected. This pitfall is listed to preempt a reviewer concern rather than to flag a real risk.

### Pitfall 6: Lang-agnostic contract assertion over-fires for DE/ES/FR/EN

**What goes wrong:** The proposed assertion `vocab.freq instanceof Map && vocab.freq.size > 0` (if applied in the browser adapter rather than the fixture runner) would fail for DE/ES/FR/EN because those languages don't ship a freq sidecar -> `state.freq` is an empty Map -> size 0.

**Why it happens:** The fixture runner's Pitfall-2 assertion (lines 118-120 of `check-fixtures.js`) is correctly scoped to NB/NN only. A blanket assertion in the adapter would over-fire.

**How to avoid:**
- Keep the contract assertion static (grep the source for the field being present in the literal) — language-agnostic, doesn't care about runtime state.
- If a dynamic assertion is added, gate on `if (lang === 'nb' || lang === 'nn')`.

**Warning signs:** Adapter-side assertion fires on every non-Norwegian page load, spams console.warn. Reproducer: load extension, switch language to EN, type in a textarea, watch for warnings.

### Zipf term inside the rule (unchanged, verification-only reference)

```javascript
// extension/content/spell-rules/nb-typo-fuzzy.js — lines 56-71 (no changes)
function scoreCandidate(query, cand, d, vocab) {
  const pref = sharedPrefixLen(query, cand);
  const suff = sharedSuffixLen(query, cand);
  let s = pref * 15 + suff * 10 - d * 100;
  if (cand.length < query.length) s -= 50;
  if (isAdjacentTransposition(query, cand)) s += 40;

  // Zipf tiebreaker — vocab.freq is hydrated from freq-{lang}.json by
  // vocab-seam-core.buildIndexes (Phase 3-01). Empty Map for languages
  // without a sidecar (de/es/fr/en) — fine, fuzzy is NB/NN-only anyway.
  if (vocab && vocab.freq) {
    const z = vocab.freq.get(cand);
    if (typeof z === 'number') s += z * ZIPF_MULT;
  }
  return s;
}
```
