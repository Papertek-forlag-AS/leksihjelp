## Decisions Made

- **Empty-Map default for getFreq():** Mirrors the Set/Map precedent of `getValidWords` / `getNounGenus` etc., not the null-returning `getBigrams` style. Rationale: nb-typo-fuzzy's null-guard is `if (vocab && vocab.freq)`; an empty Map is truthy so the guard passes, then `vocab.freq.get(cand)` returns undefined for DE/ES/FR/EN (no sidecar shipped), and the inner `typeof z === 'number'` check cleanly skips the Zipf boost. No crash, no special-casing in rule code. (Pitfall 1 in research.)
- **Inline regression guard, not a new release-gate script:** Research Open Question 1 closed in favour of a one-line assertion inside `check-fixtures.js` main() rather than spinning up `scripts/check-adapter-contract.js`. Pattern matches Plan 03-01 Pitfall 2 (fail-loud sanity check at the top of `main()`). Cheaper to ship, one test run per release sweep, easy to extend if more adapter-shape invariants surface.
- **Static source-regex, not dynamic runtime assertion:** A dynamic `vocab.freq.size > 0` check would false-fire on DE/ES/FR/EN where no freq sidecar legitimately ships. The defect is about the adapter literal dropping a field, so source-level regex is the correct contract target. (Pitfall 6 in research.)
- **Loose whitespace regex (`freq:\\s*VOCAB\\.getFreq\\(\\)`):** Alignment is a stylistic choice; tightening the regex to match exact column alignment would make a legit code reformat trip the gate. Loose regex protects the semantic contract without codifying style.
