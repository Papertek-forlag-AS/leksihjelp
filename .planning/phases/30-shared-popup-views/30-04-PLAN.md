# Plan 30-04 — Lockdown sidepanel UX integration

## Why this plan exists

Phase 30 Plan 02 landed the leksihjelp sidepanel mount in the lockdown
writing-environment, and Plan 01 sub-step E (executed 2026-04-29) shipped
`popup-views.css` so the views render with proper styling. Staging UAT then
surfaced three integration gaps that are out of scope for "make the panel
mount and render" but are blockers for shipping leksihjelp-spellcheck as a
real teacher-selectable resource profile in lockdown:

1. **Three language pickers competing for the same fact.** The leksihjelp
   floating "Aa" widget, the leksihjelp panel's language pills, and lockdown's
   own writing-environment language selector all claim ownership of "what
   language the student is writing in." When the leksihjelp-spellcheck engine
   is the selected resource profile, leksihjelp's picker must be the single
   source of truth — driving lockdown's spellcheck language and its own
   dictionary direction.
2. **The Aa floating widget is draggable in lockdown** (extension-style),
   which lets it overlap the sidepanel chrome (see screenshot
   `lockdown/local/screenshots/2026-04-29 at 10.56.44.png` — the yellow Aa hand
   sits on top of the sidepanel's "Innstillinger" toggle). In lockdown the
   widget should be **pinned**.
3. **Clicking the Aa cycles language today**, but the more useful action in
   lockdown is the right-click-equivalent: open the leksihjelp lookup /
   surface the panel for the currently-selected text. Cycling has no value
   when the dedicated picker is right there.

A larger architectural follow-up landed as a tactical fix in 2.8.2 but
deserves a proper home here:

- **Bank-flatten + dictionary state population.** `popup.js` populates
  `viewState.dictionary`, `viewState.allWords`, `viewState.inflectionIndex`,
  `viewState.nounGenusMap`, etc. before mounting `dictionary-view`. The
  lockdown sidepanel host (`leksihjelp-sidepanel-host.js`) inlines a minimal
  `flattenBanks` + `BANK_TO_POS` + `genusToGender` to mirror this — so
  searches return results and the language switcher / direction labels
  refresh on first paint. The full set of derived indexes (inflection,
  noun-genus, NB enrichment) is NOT populated. Plan 30-04 should extract
  these helpers into `extension/popup/dict-state-builder.js` (or similar),
  expose `buildDictState({ raw, sisterRaw, lang, uiLang })`, and have both
  popup.js and the lockdown host import it. Mirror the dep-injection
  contract used for the view modules so `check-popup-deps` stays green.

Plus two smaller follow-ups discovered on the same screenshots:

- **Direction toggle stuck on "ES"** — `leksihjelp-sidepanel-host.js` lines
  87–91 hard-code `<span class="target-lang-code">ES</span>` in the scaffold.
  When dictionary-view's language-change handler fires, it should rewrite
  these spans, but evidently the lockdown initial render shows "NO → ES"
  even with no Spanish ever selected. Either the rewrite isn't running or
  the `currentLang` hand-off is wrong on first paint. Bundle the fix here.
- **Grammar features list in the settings drawer** — `settings-view.js`
  intentionally does NOT handle grammar features (the wiring is host-owned
  in `popup/popup.js` because it needs the extension's vocab-store / fixture
  registry). Plan 30-02 punted on this and the 30-01-E follow-up dropped the
  empty block from the scaffold. Port the grammar-features-list wiring into
  the lockdown sidepanel host (or factor it out of popup.js into a shared
  helper that both popup.js and the lockdown host can call) so teachers
  configuring exam profiles can keep the toggleable grammar surfaces.

## Goal

When the **leksihjelp-spellcheck** resource profile is the selected engine in a
lockdown writing test:

- One language picker (leksihjelp's) drives both spellcheck and dictionary.
- Lockdown's own writing-environment language selector is hidden (or
  read-only, mirroring leksihjelp's state) for the duration of the test.
- The Aa floating widget is rendered at a fixed position, not draggable.
- Clicking the Aa opens the leksihjelp sidepanel / activates the same
  surface that right-clicking the Aa in the extension popup opens.
- Direction toggle in the panel reflects the active language pair.
- Settings drawer renders grammar-features again, fully wired.

When a non-leksihjelp engine is selected, the lockdown writing-environment
picker remains the source of truth and the leksihjelp UI stays inert (already
the case via the `bootLeksihjelp` gate in `leksihjelp-loader.js`).

## Acceptance criteria

- [ ] Selecting a target language in the leksihjelp panel updates lockdown's
      writing-environment spellcheck language without round-tripping through
      the user's manual click on lockdown's own picker.
- [ ] Vice versa: any path that would have set lockdown's writing language
      (initial test config, teacher mid-test override) flows through the same
      single source so the leksihjelp panel stays in sync.
- [ ] Lockdown's own language picker is hidden or visibly read-only when the
      `leksihjelp-spellcheck` profile is active. (Decision in research: hide
      vs. read-only-mirror.)
- [ ] The Aa floating widget renders at a fixed editor-relative anchor, has
      no drag affordance (cursor not `grab`/`grabbing`, no `mousedown` drag
      handler attached), and does not overlap the sidepanel chrome at any
      viewport width >= 1024px.
- [ ] Clicking the Aa surfaces the same UI as right-clicking it in the
      extension. (Research: confirm what right-click does — open popup vs.
      open Settings vs. open lookup card. The user's mental model is "open
      the dictionary panel," but the extension's actual behavior must be
      verified before the rebind is implemented.)
- [ ] Direction toggle (`#dir-no-target`, `#dir-target-no`) shows the actual
      target language code, not "ES". Tested by switching among NB/NN/EN
      and verifying the `.target-lang-code` spans update.
- [ ] Settings drawer's grammar-features list renders for NB at minimum
      (the engaged-by-default grammar features), with toggles wired through
      to the spellcheck engine the same way the extension popup does.
- [ ] Cross-process sync verified: leksihjelp's `LANGUAGE_CHANGED` runtime
      message propagates through lockdown's chrome shim from the
      floating-widget script to the sidepanel host (or, if the shim is
      one-way, document the gap and fix it explicitly here).

## Research findings (resolved 2026-04-29)

1. **Right-click on Aa**: floating-widget.js has no contextmenu handler. The
   "Aa pill" the user sees in lockdown screenshots is actually
   **lockdown's own** `editor-toolbar.js:initSpecialCharsPanel()` —
   a writing-environment-native sticky pill bottom-right. NOT leksihjelp's
   widget. Inside leksihjelp's actual floating-widget the language picker
   is `.lh-lang-toggle` rendered inside the lookup card, with a drag handle
   on `.lh-header`. There is no right-click — the "right-click sets
   spellcheck language" mental model the user described actually corresponds
   to clicking the language buttons inside `.lh-lang-toggle`. So
   "click Aa = right-click in extension" decomposes to: clicking the small
   Aa pill should open the lookup card, where the language pills are visible.

2. **Lockdown's writing-env language picker**: `editor-toolbar.js:initSpecialCharsPanel()`.
   `selectLanguage(groupId)` does:
   - `editor.setAttribute('lang', ...)` for native browser spellcheck
   - `window.__lexiVocab?.loadLanguage?.(groupId)` for leksihjelp
   - Does NOT broadcast `LANGUAGE_CHANGED` — sync is one-way (lockdown → leksihjelp)

3. **Chrome shim message router**: needs verification per task; the loader
   exposes `host.chrome.runtime.sendMessage` but the listener side may or
   may not propagate to other scripts. Will be tested as part of Task 1.

4. **Grammar-features wiring location** in popup.js: ~lines 600–650 (renders
   into `#grammar-features-list`). Reads from `extension/data/grammarfeatures-{lang}.json`
   and from `chrome.storage.local.enabledGrammarFeatures`. Extractable into
   `extension/popup/grammar-features-section.js` with the same dep-injection
   contract as the other view modules.

5. **Special-char panel coupling**: the lockdown language picker is the entry
   point for the SPECIAL_CHAR_GROUPS char panel (é/ñ/ü side panel). Hiding
   the prompt removes that surface too — Task 1 must split the language-pick
   concern from the special-char-display concern, or have the Special Char
   panel listen to LANGUAGE_CHANGED.

## Tasks

### Task 1 — Single-source language picker (lockdown ⇄ leksihjelp)

When `leksihjelp-spellcheck` resource profile is active:

- Hide lockdown's "Aa Språk?" prompt + picker (the sticky bottom-right pill).
  Special-char side panel still appears, driven by the active language event.
- Leksihjelp's panel + floating-widget pills become the sole pickers.
- Wire two-way sync via `LANGUAGE_CHANGED`:
  - Leksihjelp panel picks lang → broadcast `LANGUAGE_CHANGED` (already does
    this; verify it reaches editor-toolbar through the chrome shim).
  - editor-toolbar listens for `LANGUAGE_CHANGED` → applies
    `editor.setAttribute('lang', ...)` + special-char panel rebuild +
    persists into lockdown's writing-env state.
- When the resource profile is NOT leksihjelp, lockdown's picker stays as-is.

Acceptance: switching language in leksihjelp's panel makes a native
browser-spellcheck red squiggle appear under a misspelled word in the
matching language without the student having to also touch the lockdown
picker.

### Task 2 — Pinned Aa + click-to-open-card

In lockdown context, the floating-widget's small "Aa" pill (its
collapsed-form trigger) should be:

- Anchored at a fixed editor-relative position (no drag).
- Click → opens the lookup card (which already renders `.lh-lang-toggle`
  with language pills). The lookup card itself stays draggable inside the
  editor pane but doesn't escape it.

Acceptance: at viewport ≥ 1024px, the Aa pill never overlaps the
sidepanel chrome. Drag affordance gone (no `cursor: grab/grabbing`,
no mousedown drag handler attached in lockdown context).

### Task 3 — Direction-toggle binding fix

Confirm the `target-lang-code` rewrite path runs in the lockdown sidepanel
host on initial paint (already triggered by the populateDictState +
`updateLangLabels` call shipped in 2.8.2 — verify it actually fires for
all three languages NB / EN / NN).

### Task 4 — Shared `dict-state-builder.js`

Extract `flattenBanks`, `getTranslation`, `generatedFromRefs`,
`norwegianInfinitive` (incl. irregular-verbs map), `BANK_TO_POS`,
`genusToGender`, `bankToPos` from popup.js + the lockdown host's
inline copies into `extension/popup/dict-state-builder.js`. Both popup.js
and the lockdown sidepanel host import it. `check-popup-deps` stays green.

### Task 5 — Shared `grammar-features-section.js`

Extract grammar-features rendering loop from popup.js into
`extension/popup/grammar-features-section.js`. Sidepanel host re-renders
the settings drawer with the grammar block restored, `showSection.grammar = true`.

## Original research questions (now resolved above)

1. What does right-click on the Aa do in `floating-widget.js`? (Open popup?
   Open settings? Open lookup?) The plan's "click=right-click" rebind
   depends on the answer.
2. Where is lockdown's own writing-environment language picker rendered?
   What state owns it, and how is the spellcheck language plumbed from
   that state into the lockdown spellcheck engine?
3. Does lockdown's chrome shim (`public/js/leksihjelp-loader.js` plus the
   message-router elsewhere) propagate `runtime.sendMessage` events from
   the floating-widget host to the sidepanel host? If not, the sync layer
   needs an explicit bridge.
4. Where is the grammar-features rendering loop in `popup/popup.js`? Can it
   be factored out into a shared module under `extension/popup/` (e.g.
   `popup/grammar-features-section.js`) that both the popup and the
   lockdown sidepanel host import? Constraints: same dep-injection contract
   as the existing view modules (per `check-popup-deps`).

## Out of scope

- **Skriveokt-zero parity** (was Plan 30-03, now deferred to a later phase
  per user instruction 2026-04-29 — `leksihjelp-zero` work moves out of the
