Audit and harden the Side Panel wiring so the panel persists across tab switches on macOS:

1. **manifest.json** — confirm `"side_panel": { "default_path": "popup/popup.html" }` and the `"sidePanel"` permission are present. Already present per current state — no change unless the audit reveals otherwise.

2. **service-worker.js** — add (if not present) the once-at-install setup:
   ```js
   chrome.runtime.onInstalled.addListener(() => {
     if (chrome.sidePanel?.setPanelBehavior) {
       chrome.sidePanel.setPanelBehavior({ openPanelOnActionClick: false })
         .catch(err => console.warn('setPanelBehavior failed:', err));
     }
   });
   ```
   This call is required for the panel to stay open on tab switch instead of being scoped to the action-click flow. (Without it, the panel's lifecycle is tied to per-tab options and will appear to "close" on tab switch on some macOS Chrome builds.)

3. **popup.js Fest handler** — keep the existing fallback to detached window for browsers without `chrome.sidePanel`. The current code at ~line 2544 is correct in shape; only verify after the service-worker change that `chrome.sidePanel.open({ tabId })` succeeds and the panel sticks across tab switches.

4. Hide-in-side-panel detection: the existing check (`window !== window.top`) is brittle. Side Panel renders top-level, not iframed — the check might be wrong. Replace with a more reliable signal: the URL search-param `?context=sidepanel` set by service-worker before opening, OR detect via `chrome.runtime.getContexts({ contextTypes: ['SIDE_PANEL'] })`. Pick whichever is simpler to verify in the browser.

Do NOT change unrelated popup behavior. The goal is to make POPUP-02's browser test in Task 2 pass.
