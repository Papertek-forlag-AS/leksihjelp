<objective>
Ship a Node build script that stream-parses the NB N-gram 2021 `digibok-bigram.csv.gz` (7.15 GB gzipped), derives weight-bucketed NB and NN bigrams matching the existing `{prev: {next: weight}}` schema, merges them against the hand-authored bigrams already in `extension/data/bigrams-{nb,nn}.json` using a max-merge strategy so curated idioms survive, and writes the expanded files back to disk within a 50 KB gzipped per-file cap. The current `bigrams-nb.json` ships with only 72 head-word entries (2.6 KB raw, 825 B gzipped); Phase 3's word-prediction ranking work (WP-02) cannot produce useful context-aware suggestions on that corpus — DATA-03 grows it by an order of magnitude while staying well inside the bundle-size budget.

Purpose: The existing bigram corpus is function-word-only (`det`, `jeg`, `hun`); Phase 3 ranking improvements depend on content-word coverage. By deriving top-K continuations per predecessor from a 580k-book corpus under CC-0 license, we get Norwegian collocations at real-world frequencies. By max-merging with the hand-authored file, we keep curated idioms (`først og fremst`, `i stedet for`, `på grunn av`) at their intended strong-association weight of 3 even when corpus statistics alone would rate them 2. This is the Pattern 5 / Pitfall 7 discipline from the research.

2. `async function ensureCorpus()` — identical shape to Plan 02-01's ensureCorpus but for the bigram URL. 3. `async function streamBigrams({ targetCorpusLang })` — opens `createReadStream(CORPUS_FILE).pipe(zlib.createGunzip())`, wraps in readline, iterates every line. For each: `parts = line.split('\t')`. Skip if `parts.length < 4`. `[first, second, lang, freqStr] = parts` (note: bigrams have FOUR data columns before the json column — `first`, `second`, `lang`, `freq`). Skip if `lang !== targetCorpusLang`. Lowercase both first and second. Reject either if `!WORD_RE.test(word)`. Parse freq, skip if `!Number.isFinite(freq)` or `freq < MIN_BIGRAM_FREQ`. Accumulate in a `Map<string, Map<string, number>>`: `bigramCounts.get(first).set(second, (bigramCounts.get(first).get(second) ?? 0) + freq);`. Also compute per-first-word total (`firstTotals.get(first) += freq`) for ratio bucketing in the next step. Log progress every 20M lines. Return `{ bigramCounts, firstTotals, totalRows }`.

4. `function bucketize(pairFreq, firstTotal, grandTotal)` — quantises continuous bigram frequency into the integer buckets {1, 2, 3} per Pattern 5. Rule: compute `ratio = pairFreq / expectedIfIndependent`, where `expectedIfIndependent = firstTotal * (nextUnigramFreq / grandTotal)` approximated by `firstTotal / numDistinctSeconds(first)` — for Phase 2 simplicity we use the simpler rule from the research:
   - `ratio = pairFreq / firstTotal` (how concentrated is this continuation within all continuations of `first`).
   - If ratio > 0.05 (this continuation takes ≥5% of all continuations of `first`): weight = 3 (strong association).
   - If ratio > 0.015: weight = 2.
   - Else: weight = 1.
   
   This is the concrete threshold Phase 2 uses; Phase 3 WP-02 may retune — do not over-engineer now.

5. `function deriveBigrams({ bigramCounts, firstTotals, validWords, topPredecessors, topContinuations })` — ranks predecessors by their total count (sort `firstTotals` desc), takes top `topPredecessors`, for each takes top `topContinuations` continuations by pair frequency. Filters: `validWords.has(first)` and `validWords.has(second)` (word-list intersection; predecessors with spaces are impossible here since bigram source is word-pair-tokenized). Returns `{ first: { second: weight } }`.

6. `function mergeBigrams(existing, derived)` — the Pitfall 7 guard. Implementation:
   ```javascript
   const out = JSON.parse(JSON.stringify(existing));  // deep clone, preserve structure
   for (const [first, nextMap] of Object.entries(derived)) {
     if (!out[first]) out[first] = {};
     for (const [second, w] of Object.entries(nextMap)) {
       out[first][second] = Math.max(out[first][second] ?? 0, w);
     }
   }
   return out;
   ```
   Multi-word prev-keys in `existing` (space-containing keys like `"i stedet"`) pass through untouched because `derived` has no space-keys to collide with them. Single-word prev-keys with hand-authored weight 3 stay at 3 if the derived rule only rates them 2 — max wins.

7. `function enforceBudget(obj, maxGzipBytes, opts)` — if over budget: reduce `topContinuations` by 1 (from 8 → 7 → 6 … down to min 4) and re-run deriveBigrams+mergeBigrams. If still over at `topContinuations=4`, reduce `topPredecessors` by 200 and retry. If still over at 500 predecessors × 4 continuations, throw — the research's budget target becomes impossible, and the planner needs to reconsider. Log each backoff step: `"bigrams-nb.json over budget (52,341 gz bytes), reducing topContinuations: 8 → 7"`.

8. `function writeBigrams(outPath, obj)` — sort top-level keys alphabetically for idempotency, use `JSON.stringify` without whitespace. Optional: also sort inner objects by (weight desc, next asc) for readability — test that this stays valid per the `normalizeBigrams` contract before committing to it; if sorting breaks any consumer assumption (unlikely — it iterates via `for...in`), fall back to alphabetical inner keys.
