# Phase 05.1 Plan 03: Gender-Rule Three-Beat Explain Copy (Gap C) Summary

**Upgraded nb-gender explain from two-beat ("Et kan være feil kjønn — prøv En.") to three-beat ("Et kan være feil kjønn — by er hankjønn. Prøv En."), adding noun_display + actualGenus finding fields, i18n gender-class labels (NB hankjønn/hunkjønn/intetkjønn; NN hankjønn/hokjønn/inkjekjønn), and a reusable getString(key, lang) helper on __lexiSpellCore.**

## Accomplishments

- Three new i18n keys per locale block (9 lines total across NB/NN/EN) naming the canonical gender classes using Språkrådet-standard Norwegian grammar terms.
- Reusable `getString(key, lang)` helper on `__lexiSpellCore` — reads `__lexiI18n.STRINGS[lang][key]` directly (bypasses `__lexiI18n.t()` which routes via UI locale). Designed for rule `explain()` callables that produce both NB and NN strings in a single call, routed by document lang rather than UI lang.
- `nb-gender.check()` now emits `noun_display` (cased noun form like `Byen`) + `actualGenus` (`m`/`f`/`n`) on every gender finding; `nb-gender.explain()` consumes them via `GENUS_TO_LABEL_KEY` map to produce three-beat copy.
- Defensive fallback to Phase 5 two-beat copy when the new fields are missing — honored by `check-explain-contract`'s fake-finding path (which omits both fields) and preserves the `{nb, nn}` non-empty-strings contract.
- Phase 4 feminine-as-common-gender acceptance (NB `en bok` is NOT flagged) preserved — `check()` logic unchanged, only the emit block grew.
- All 6 release gates green: `check-explain-contract` 5/5, `check-explain-contract:test` PASS, `check-fixtures` 281/281, `check-network-silence` PASS, `check-bundle-size` 10.14 MiB / 20 MiB cap (9.86 MiB headroom — +30 lines of code + 9 i18n keys is ~1 KB, way below cap).
- `COPY-REVIEW.md` gender row updated with three-beat templates + 6 worked examples (NB/NN × m/f/n); new 2026-04-21 re-review log entry with dyslexia-persona proxy PASS outcome; forward-looking tone-check sketch of Plan 05.1-04 dialect-mix templates for executor continuity.

## Resolution of Open Questions from 05.1-RESEARCH.md

**Open question #3 — explain signature:** **Option A chosen.** The rule emits two new fields (`noun_display`, `actualGenus`) on every gender finding; `explain(finding)` reads them and routes to `getString(key, 'nb')` / `getString(key, 'nn')` for per-register label lookup. The `{nb, nn}` contract is unchanged. No modification to the finding base shape required beyond additive fields (which the fixture runner's 4-field match tolerates). Discarded Option B (change explain signature to accept lang) because it would churn all 5 popover-surfacing rules + the renderExplain contract; Option A contains the change to the one rule that needs it.
