# Phase 02.1: Close SC-4 bundle-size cap - Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Close Phase 2 Success Criterion #4 (bundle-size ceiling) so `npm run check-bundle-size` exits 0 and GitHub Releases can ship again. Phase 2 already landed the gate and JSON minification at package time; the current zip is 10,599,772 bytes (10.11 MiB), 114 KB over the self-imposed 10 MiB ceiling.

This phase closes the gap by **raising the ceiling to 20 MiB** and correcting internal docs that over-framed the ceiling as a "publicly-stated promise" (it never was — the landing page does not mention a size number). Nothing user-visible changes, no feature is removed, no vocab or audio data is refactored.

Out of scope: English-on-demand refactor, audio lazy-fetch, data-trim efforts. Those remain as optional future moves only if bundle size becomes a real problem again.

</domain>

<decisions>
## Implementation Decisions

### Remediation path
- **Raise `CEILING_BYTES` from `10 * 1024 * 1024` to `20 * 1024 * 1024` in `scripts/check-bundle-size.js` and `scripts/check-bundle-size.test.js`.** Current zip (10.11 MiB) then passes with ~9.9 MiB headroom — comfortably accommodates Phase 3-5 data/rule growth without thrashing the ceiling decision mid-milestone.
- **Keep `check-bundle-size` as a HARD gate at the new ceiling.** Script still runs in the Release Workflow and still exits 1 if over. Purpose shifts from "enforce 10 MiB promise" to "catch accidental growth regressions" (e.g. pretty-printed data checked in, huge new asset added silently).
- **No changes to the extension source** — no BUNDLED_LANGUAGES edits, no data files removed, no audio plumbing changes. The 20 MiB number is set by this phase and the existing zip is already under it; there is nothing further to shrink in 02.1.

### Docs correction (ceiling framing)
- **Drop the "publicly-stated promise" language everywhere it appears.** Replace with "internal engineering ceiling" or equivalent neutral phrasing. Call sites found:
  - `CLAUDE.md:350` (Release Workflow note)
  - `scripts/check-bundle-size.js:9` (header comment) and `scripts/check-bundle-size.js:150` (failure-mode console output)
  - `.planning/PROJECT.md:52, 79` (bundle-size risk + technical-constraints references)
  - `.planning/ROADMAP.md:42, 49` (Phase 2 goal + SC-4 wording)
  - `.planning/REQUIREMENTS.md:23` (DATA-03 "~10 MB ceiling")
- Goal: future phases inherit an honest description of the constraint. The 10 MiB number was never verified on `backend/public/index.html`; grepping for `10\s*M[iB]?B?` on the landing page returns zero hits.
- Historical plan/summary documents under `.planning/phases/02-data-layer-frequency-bigrams-typo-bank/` that repeat the phrasing are historical record — leave untouched so the archive stays faithful to what was decided at the time.

### SC-4 rewording
- The existing SC-4 reads "stays within the 10 MB ceiling" and MUST be rewritten to match the new number. Planner/executor picks the exact wording; baseline direction: "Packaged zip from `npm run package` stays under 20 MiB, verified by `scripts/check-bundle-size.js` exit 0." Phase 2.1's completion requires this line updated in `ROADMAP.md` and any parallel place it appears in `REQUIREMENTS.md` DATA-03.

### Gate behavior
- `scripts/check-bundle-size.js` still: runs `npm run package`, measures the zip, prints per-directory breakdown, exits 0/1 against the new threshold. Output messaging updated to drop the "publicly-stated promise" framing.
- Test file `scripts/check-bundle-size.test.js` (Test 4/5 in particular — "behavior: exit code matches reported bytes vs 10 MiB cap") needs its constant AND its test name updated to the new ceiling.

### Fact corrections documented for future reference
- **Chrome Web Store has no 10 MB limit.** The effective upload ceiling is 2 GB. There was never a store-side reason for the cap.
- **The 10 MB number was internal only.** Originated in `PROJECT.md` technical-constraints and propagated into CLAUDE.md, the check-bundle-size script, and subsequent plans. Phase 2.1 corrects that lineage.
- **The BUNDLED_LANGUAGES mechanism already supports on-demand language downloads** (de/es/fr use it). If bundle size becomes a real problem later, moving English out of bundled → on-demand is a known cheap refactor (see Deferred Ideas).

### Claude's Discretion
- Exact wording of the corrected "internal engineering ceiling" prose — as long as it drops the "publicly-stated promise" claim and is accurate.
- Exact SC-4 rewording in ROADMAP.md (baseline direction above; polish to match roadmap voice).
- Whether to include a one-line comment in `check-bundle-size.js` briefly noting the 2026-04-19 raise + rationale, or to leave the ceiling change clean. Either is fine.

<specifics>
## Specific Ideas

- **Keep the gate, raise the bar.** The gate's long-term value is regression detection — "did someone accidentally check in a pretty-printed JSON" or "did a new asset balloon the zip silently." A bumped ceiling still provides that, without forcing a premature feature-removal decision to hit an unjustified number.
- **Don't re-litigate remediation choices.** The 02-05-AUDIT.md enumerated English-removal, audio-strip, and conjugation-trim paths in detail. Phase 02.1 explicitly chooses NOT to pursue any of them; the evidence is preserved so a future phase can pick one up if needed.

</specifics>

<deferred>
## Deferred Ideas

Preserved for future phases if bundle size ever becomes a real constraint (e.g. Chrome Web Store tightens limits, or Phase 3+ data additions push past 20 MiB):

- **Move English vocab to on-demand download** — 6-step refactor from `.planning/phases/02-data-layer-frequency-bigrams-typo-bank/02-05-AUDIT.md` § "What a CLEAN path would have required", but keeping the English feature: flip `'en'` out of the 4 BUNDLED_* Sets, change `stored.language || 'en'` fallbacks to `'nb'`, drop `'en'` from `scripts/sync-vocab.js:414`. English downloads on first selection like de/es/fr already do. Saves ~2.8 MiB compressed.
- **Strip bundled `audio/de/*.mp3`, fetch-on-first-play with IndexedDB cache** — 7.67 MiB uncompressed, ~3 MiB in zip. Breaks offline-first-play for German TTS of uncached words; acceptable if the fetch is backgrounded.
- **Dedupe `audio/de/*` against ElevenLabs premium output** — premium users don't need the offline pack; could ship a smaller free-tier audio set.
- **Trim rarely-used noun/verb conjugation branches in `data/de.json`** (8.5 MB source, largest single file) — requires production-lookup telemetry we don't have; high design cost.
- **Remove English vocabulary entirely** — the "Path 1" from 02-05-AUDIT.md. Documented cleanup sequence available, but the landing page currently advertises English (`backend/public/index.html:7, 558`) so would require copy updates AND a user-facing feature-removal decision. Only appropriate if English usage proves negligible and the team wants to reclaim ~4.5 MiB.

Unchanged carry-forward blockers from Phase 2 (NOT Phase 02.1's responsibility):
- NN phrase-infinitive triage (~214 entries in sibling `papertek-vocabulary` verbbank.json) — standalone sibling-repo cleanup PR.
- Missing `fin_adj` entry in NB + NN adjective banks — sibling-repo data gap.
