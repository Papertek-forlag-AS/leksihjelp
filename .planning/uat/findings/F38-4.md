# F38-4: P3-hint dotted underline not visually rendering in some contenteditable hosts

## Reproduction

1. Sideload extension v2.9.21 (with `border-bottom: 3px dotted #f59e0b` for P3 hints).
2. Open `skriv.papertek.app/#/doc/<any>` (which has its own embedded leksihjelp).
3. Set popup foreign-language: French. NB locale.
4. Type a fr-aspect-hint trigger: `Hier je mangeais une pomme.`

**Observed output:**

```
- [lexi-spell] check {lang: 'fr', vocabSize: 18920, findingsCount: ≥1}
- [lexi-spell] markers rendered {rendered: ≥1, skipped: 0}
- Marker IS in DOM (confirmed by sequential-correction navigation: clicking
  through corrections lands on the fr-aspect-hint finding; Lær mer expands
  correctly; popover with explain.nb renders).
- BUT no visible underline on the trigger token (e.g. `mangeais`).
- The 3px amber dotted border-bottom is not painted, or is painted in a layer
  the user can't see.
```

**Expected output:**

```
- 3px amber dotted underline visible beneath `mangeais` (or similar trigger).
```

## Root cause hypothesis

Three candidates:
1. **Two leksihjelp instances on the same page** — skriv.papertek.app has its
   own embedded leksihjelp via the lockdown sync-style mechanism (per memory
   `project_lexi_in_skriv_integration.md`). The embedded version's overlay
   may win positioning/z-index and override the extension's CSS, OR uses an
   older v2.9.x without the amber bump.
2. **Tailwind contenteditable interaction** — the editor uses
   `text-stone-800 dark:text-stone-200 leading-relaxed`; the `leading-relaxed`
   line-height may visually clip the 3px border-bottom against the next line.
3. **Overlay positioning** — `#lexi-spell-overlay` may render the marker as
   an overlay positioned over the text, but the underline's vertical position
   doesn't account for the editor's line-height correctly. The functional
   click-through works because the marker has hit-testing geometry, but the
   visual underline is mis-positioned.

Most likely: **(1)** — testing on a clean page (Google search box, plain
textarea) where only the sideloaded extension runs would isolate this.

## Why this isn't blocking

- The rule fires correctly (`findingsCount: ≥1`, marker rendered).
- The popover renders correctly with explain.nb text.
- The Lær mer panel expands correctly with pedagogy data + examples.
- Sequential-correction navigation lands on the finding.
- The user gets all the functional value of the rule; they just can't see
  the entry-point dot on this specific host.

## Fix candidates (deferred to future plan)

- Confirm reproduction on a clean page (no skriv-embedded leksihjelp). If
  underline shows there, the fix is upstream in skriv to resync its embedded
  leksihjelp. Path: skriv runs its own version-pin update.
- If underline is missing on clean pages too: investigate marker geometry
  in `extension/content/spell-check.js` rendering — possibly bump z-index,
  add `display: inline-block` or position adjustment, or move the visual
  cue from `border-bottom` to `text-decoration: underline`.
