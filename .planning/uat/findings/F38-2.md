# F38-2: NN locale partial — button labels translate, popover explanation body stays in NB

## Reproduction

1. Set popup språk → nynorsk. Reload page.
2. Trigger a spell-check popover that surfaces a "Lær mer" link.

**Observed output:**

```
- "Lær mer" link label correctly renders as "Lær meir" (nynorsk).
- But the explanation body inside the popover/Lær meir panel is still in bokmål.
- Other UI strings observed in NB while popup language is NN (specific strings TBD —
  walker did not enumerate; flag for the deeper Phase 38-04 NN/EN locale walk to capture).
```

**Expected output:**

```
- All user-facing strings in popover, explanation body, and surrounding chrome render in nynorsk
  when popup språk = nynorsk.
- OR: documented carve-out — some explanations don't exist in nynorsk and fall back to bokmål
  by design (this would still warrant a per-rule register-coverage map so we know which rules
  have NN strings and which don't).
```

## Root cause hypothesis

Three possibilities, not yet triaged:
1. Per-rule `explain.nn` strings exist for some rules but not others — fallback returns `explain.nb`. Confirm by enumerating rules under `extension/content/spell-rules/` and checking each has both `nb` and `nn` registers (UX-01 / `check-explain-contract` already enforces this for popover-surfacing rules — so if it's a popover rule, the NN should exist; possibly the rule firing here is not in the popover-surfacing TARGETS list).
2. i18n keys for popover chrome (titles, button helpers) covered, but body content (the actual explanation paragraphs) is rendered from a different source (pedagogy data via papertek-vocabulary) that doesn't carry NN registers.
3. Locale not propagating from popup to content-script seam — the i18n lookup is happening with `lang='nb'` even when popup says `nn`.

Walker (Geir): "Not a blocker, we should just do that in a later phase. Maybe the explanations are not existing in nynorsk. I also see other strings in bokmål when in the nynorsk-UI. But no problem."

**Why slipped past gates:** `check-explain-contract` validates that `rule.explain` returns `{nb, nn}` for the 5 popover-surfacing rules (UX-01) — this catches the rule layer. It does NOT catch:
- Pedagogy data NN coverage (sourced from papertek-vocabulary, not the rule files).
- Generic UI string i18n coverage outside the popover-surfacing TARGETS.
- Live locale propagation from popup → content script.

This is exactly the kind of gap a real-browser walk surfaces that no Node gate sees.
